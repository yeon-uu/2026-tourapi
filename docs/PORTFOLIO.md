### 2-1. AI-DO (2026-tourapi) — 풀스택 / 솔로 🏆

전국 기차역을 랜덤으로 뽑아 그 지역의 테마형 로컬 미션으로 바꾸고, 방문할 때마다 스탬프로 아카이빙하는 AI 웹 서비스. **인구감소 위험 지역에 가중치**를 둬 지방 소멸 대응이라는 정책 목적을 녹였습니다.

- **공모전**: 관광데이터랩 프롬프톤 — **파이널리스트(574팀 중 결선 15팀, 예선 통과 195팀)**
- **역할**: 기획 · 백엔드 · 프론트 **단독 구현**
- **스택**: FastAPI · PostgreSQL · Docker · Nginx · EC2 · OpenAI · TourAPI

**서비스 흐름**
역 가챠 뽑기 → AI(엔노이아/두두)가 역 주변 관광데이터로 현장 미션 생성 → 사용자가 미션 완료 체크 → 기념 스탬프 카드 발급 → 노선별 컬렉션 아카이빙

**스크린샷**
- `[이미지 1]` 역 가챠 뽑기 → 단양역 카드 → 두두 챗봇 취향 선택 흐름
- `[이미지 2]` 체크리스트 완료 → 스탬프 카드 획득 → 노선별 컬렉션(1/73역)
- `[엔노이아 챗 화면]` AI 미션 생성 대화 + 여행 성향 리포트 발급

**기술 선택 이유**
- **PostgreSQL + JSONB**: AI가 생성한 가변 구조의 미션 배열을 정규화 없이 유연하게 저장 (ADR-002)
- **모놀리식 단일 서버**: 12일·1인 규모에서 배포·디버깅 단순화, 단일 EC2로 비용 최소화 (ADR-004)
- **Vanilla JS**: 5페이지 미만 규모라 SPA 프레임워크의 빌드 파이프라인이 오히려 오버헤드 (ADR-005)

**💡 트러블슈팅 ① — 공공 TourAPI 호출 한도 & 응답 지연**
- **문제**: 같은 역의 주변 관광데이터를 미션 생성할 때마다 TourAPI에 직접 호출 → 호출 한도(rate limit) 초과 위험 + 매 요청마다 외부 API 응답을 기다려 체감 속도 저하. 외부 API는 언제든 실패할 수 있다는 점도 문제.
- **시도**: 단순 재시도만으로는 한도 문제를 못 풀고, 외부 장애 시 서비스 전체가 멈추는 구조였음.
- **해결**: TourAPI 응답을 **TTL(24h) 캐시로 래핑**해 같은 역의 중복 호출을 제거하고, 외부 API 실패 시 **fallback** 경로를 둬 서비스가 끊기지 않게 설계. ('Design for Failure' 원칙 적용)
- **학습**: 외부 의존성은 캐싱·fallback으로 **격리**해야 내 서비스의 가용성을 외부 API에 인질로 잡히지 않는다는 것을 체득.

**💡 트러블슈팅 ② — 단순 랜덤이 아닌 정책 목적의 가챠**
- **문제**: 그냥 균등 랜덤으로 역을 뽑으면 공모전 주제(지방 소멸 대응)와 연결이 약함.
- **해결**: 인구감소 위험 지역에 더 높은 확률을 부여하는 **가중치 스코어링 알고리즘**을 설계하고, 가챠 규칙(하루 5회 제한, 출발역·기보유 스탬프역 제외, 전체 수집 시 뽑기 종료)을 함께 정의.
- **학습**: 데이터(관광공사 API)와 도메인 목적을 알고리즘에 녹이면 단순 기능이 **서비스의 메시지**가 된다는 것을 배움.

**💡 트러블슈팅 ③ — 생성형 AI 출력의 보안/호환 이슈** <!-- ⚠️ 확인 필요: 강조하고 싶지 않으면 삭제 -->
- 모바일(Safari) 스탬프 사진 저장 비호환, LLM 출력 XSS 방어(저장 전 escape·`textContent` 사용), 업로드 5MB 제한·닉네임 특수문자 차단 등 **생성 시점에 보안을 적용**(사후 추가 금지)한 사례. (git 커밋 이력 근거)

### 2-2. Mo:lib — 백엔드 · 인프라 / 팀

취향에 맞는 새 콘텐츠를 찾기 어렵다는 문제에서 출발해, **영화 · 도서 · 음악을 하나의 입력으로 동시 교차 추천**하는 LLM 기반 크로스 도메인 추천 서비스. (경기대 AI컴퓨터공학부 / 소프트웨어중심대학사업단)

- **역할**: 백엔드 & 인프라 — 백엔드 토대(스캐폴딩 · CI · JWT 인증) 구축
- **스택**: FastAPI · PostgreSQL · Gemini · Docker

**스크린샷**
- `[이미지 4]` Mo:lib 아키텍처 — 외부 API(TMDB·알라딘·Spotify) → 표준 JSON 정규화 → Gemini 2.5 Flash-Lite 2-stage(감성/맥락 분석 → 도메인 추천) → 과몰입 지도 시각화

**기술 선택 이유**
- **PostgreSQL + TTL 캐싱**: 외부 콘텐츠 API 응답을 캐시해 추천 지연을 줄이고 호출 비용 관리
- **FastAPI Pydantic**: 외부 3종 API의 제각각인 응답을 표준 JSON 스키마로 정규화하는 검증 계층으로 활용

**💡 트러블슈팅 ① — 불완전한 LLM 응답이 캐시를 오염시킨 문제** *(PR #86)*
- **문제**: 음악에서 추천받으면 음악만 3개, 영화에서 추천받으면 영화만 3개처럼 **일부 도메인이 누락된 결과가 하루 종일 반복**됨.
- **원인(검증)**: Gemini가 간헐적으로 일부 도메인이 빠진 응답을 반환하는데, 이 **불완전 응답이 검증 없이 24시간 캐시에 저장**돼 같은 콘텐츠는 종일 같은 결과만 내보냄. (프롬프트를 강화해도 비결정성은 남음)
- **해결**: 기대 도메인(movie·book·music)이 **모두 채워진 응답만 캐시**하도록 가드 추가. 불완전 응답은 사용자에게 반환은 하되 **캐시는 건너뛰어 다음 요청에서 AI가 재시도**하게 함.
- **학습**: LLM은 비결정적이므로 **출력은 신뢰 대상이 아니라 검증 대상**이며, 캐시에는 '검증을 통과한 것만' 넣어야 한다는 캐시 독성(cache poisoning) 관점을 얻음.

**💡 트러블슈팅 ② — 동기 쿼리로 인한 운영 500 & asyncpg 호환** *(PR #38·#41·#30 등)*
- **문제**: `/users/me/stats` 등에서 동기 쿼리·`datetime` naive 값이 비동기 스택과 충돌해 **500 에러** 발생.
- **해결**: 해당 API를 **AsyncSession 기반으로 마이그레이션**하고, `datetime`을 **UTC aware로 변환**해 asyncpg 호환 문제를 제거.
- **학습**: FastAPI async 스택에서는 **DB 세션·시간 처리까지 비동기/UTC로 일관**되게 가져가야 한다는 것을 체감.

**💡 트러블슈팅 ③ — CD '거짓 성공(false green)' 차단** *(PR #85)*
- **문제**: `requirements.txt`가 UTF-16로 커밋돼 pip가 못 읽어 이미지 빌드가 실패하는데도, 배포 스크립트에 `script_stop`이 없어 **마지막 명령만 성공하면 CD가 초록불로 표시**돼 실패를 아무도 인지하지 못함.
- **해결**: 파일을 UTF-8로 재인코딩하고, CD에 `script_stop: true`(중간 실패 시 즉시 실패) + 타임아웃·`--force-recreate`를 추가. (RAM 문제가 아님을 OOM 0회로 확인해 불필요한 서버 증설을 차단)
- **학습**: **'초록불'이 곧 '성공'은 아니다** — 파이프라인은 실패를 실패로 보고하도록 만들어야 신뢰할 수 있다는 것을 배움.

**그 외 기여**: EC2 자동 배포(CD) 파이프라인 구축, CI에 **Gitleaks + pip-audit** 적용으로 시크릿·의존성 보안 강화.

### 2-3. Roame — 백엔드 · DevOps / 팀 (진행 중)

GPS로 하루 동선을 자동 기록하고, 그 경로를 **AI가 블로그 글로 작성**해주는 일상·여행 기록 앱.

- **역할**: 백엔드 & DevOps — 백엔드·CI를 처음부터 구축, 핵심 API 구현
- **스택**: FastAPI · 비동기 처리 · S3 호환 스토리지 · AI 서버 연동

**스크린샷**
- `[이미지 3]` (현재 버전) 날짜별 타임라인 — 지도 위 동선 핀 + 방문 장소/시간/카테고리 카드 + '글쓰기'

**구현 내용 / 기술 선택 이유**
- **비동기 블로그 생성**: AI 글 생성은 응답이 오래 걸리므로 요청을 블로킹하지 않도록 비동기로 처리
- **MinIO(S3 호환) 스토리지**: 사진·생성물 저장을 객체 스토리지로 분리
- **계약 우선 개발**: 프론트엔드 · AI 팀과 **API 스펙(계약)을 기준**으로 각자 개발

**💡 트러블슈팅 ① — 모델-마이그레이션 드리프트로 인한 운영 500** *(PR #32)*
- **문제**: init 마이그레이션 이후 **모델(코드)에는 컬럼이 추가됐지만 마이그레이션에는 누락**돼, 실제 DB에서 `daily_records` 조회 시 `map_image_url` 없음으로 **500**, `photos` insert는 `storage_key` 없음으로 실패.
- **해결**: 누락 컬럼을 정합하는 `sync_schema_drift` 마이그레이션을 신규 작성하고, **upgrade/downgrade 양방향 동작을 검증**(컬럼 추가 + storage_url→storage_key 데이터 보존 rename).
- **학습**: ORM 모델과 마이그레이션은 **자동으로 동기화되지 않는다** — 스키마 드리프트는 곧 운영 장애이며, 마이그레이션은 양방향으로 검증해야 안전하다는 것을 배움.

**💡 트러블슈팅 ② — 프론트·백엔드 API 계약 불일치** *(PR #41)*
- **문제**: 프론트가 보내는 `writing_style`(snake_case)을 백엔드가 받지 못해 **사용자의 문체 선택이 반영되지 않고**, 상세 응답에 `blog_id`가 없어 화면들 간 식별자가 일관되지 않음.
- **해결**: Pydantic 스키마에 **`writing_style` 별칭**을 추가해 프론트 요청을 그대로 수용하고, 응답에 `blog_id`를 추가해 생성·상태·상세·발행 응답의 필드를 일관화. 매핑·일관성은 테스트로 고정.
- **학습**: 팀 병렬 개발에서 **계약(필드명·타입)의 작은 불일치가 곧 기능 미반영 버그**가 된다는 것을, 별칭 수용 + 응답 일관화 + 테스트로 막을 수 있음을 체득.

**💡 트러블슈팅 ③ — AI 서버 호출 실패 격리** *(PR #13)*
- **해결**: AI 서버 호출에 **타임아웃 60초 + 최대 3회 재시도(1s→3s→5s)** 를 두되 4xx는 즉시 실패시키고, 블로그 생성 전 흐름을 **E2E 테스트 7건**으로 보호. (외부 AI는 실패한다고 가정 — Design for Failure)

## 3. 기술 (경험 중심)

> 숙련도 막대 대신, **무엇을 어디서 어떻게 써봤는지**로 적습니다.

| 기술 | 경험 |
|------|------|
| **Python · FastAPI** | 3개 프로젝트 모두에서 비동기 API 설계. Pydantic으로 외부 API 응답 검증/정규화, SSE 스트리밍, JWT 인증 구현 |
| **SQLAlchemy (async) · Alembic** | 동기 쿼리 → AsyncSession 마이그레이션으로 운영 500 해결, 모델-마이그레이션 드리프트 정합(Roame) |
| **캐싱 설계** | TTL 캐시로 외부 API 호출 한도 대응(AI-DO), 불완전 LLM 응답 캐시 가드(Mo:lib) |
| **PostgreSQL** | JSONB로 가변 구조 데이터 저장, 가중치 스코어링 기반 조회 (AI-DO) |
| **Docker · Nginx** | Docker Compose 멀티 컨테이너 구성, Nginx 리버스 프록시 + SSE 버퍼링 off |
| **AWS EC2 · GitHub Actions** | SSH 기반 자동 배포(CD), ruff/black/pytest 기반 CI 구축 |
| **보안 (CI 통합)** | Gitleaks 시크릿 스캐닝 + pip-audit 의존성 취약점 감사. 보안 동아리에서 공격자 관점 학습 |
| **외부 API 연동** | TourAPI(관광공사), TMDB·알라딘·Spotify, OpenAI/Gemini — TTL 캐시 + fallback 설계 |
